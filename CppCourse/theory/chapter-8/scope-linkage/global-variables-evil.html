    Глава VII

    Почему глобальные переменные опасны

    Если спросить опытного разработчика о главном совете по хорошему коду, скорее всего услышите: «Избегайте глобальных переменных». Речь идёт прежде всего о неконстантных глобальных переменных.

    Непредсказуемое состояние программы

    Главная проблема: любая функция может изменить глобальную переменную, и программист не всегда об этом знает:

    C++ — скрытое изменение глобальной переменной 
    #include <iostream>

int g_mode {}; // глобальная переменная

void doSomething()
{
    g_mode = 2; // тихо меняет глобальное состояние
}

int main()
{
    g_mode = 1;
    doSomething();

    // Программист ожидает g_mode == 1, но это уже 2!
    if (g_mode == 1)
        std::cout << "Угрозы нет.\n";
    else
        std::cout << "Запуск ракет...\n"; // выполнится это
    return 0;
}

    Проблема порядка инициализации

    Порядок инициализации глобальных переменных между разными файлами не определён стандартом. Если одна глобальная переменная зависит от другой из другого файла — это потенциальная ошибка:

    C++ — опасная зависимость при инициализации 
    // Файл a.cpp
int initX();
int g_x { initX() }; // инициализируется первым

int initX() { return g_y; } // g_y ещё не инициализирован!

// Файл a.cpp (продолжение)
int g_y { 5 };

// Результат: g_x == 0 (g_y ещё не был инициализирован)
    Антипаттерн: Static Initialization Order Fiasco

    Неопределённый порядок инициализации глобальных объектов из разных единиц трансляции — классическая ловушка C++. Избегайте инициализации глобальных переменных значениями других глобальных переменных из других файлов.

    Когда глобальные переменные допустимы

    Есть редкие случаи, когда глобальная переменная оправдана:

    ✅ Допустимые случаи
    • Объект существует в единственном экземпляре во всей программе
    • Используется повсеместно (например, лог-файл, генератор случайных чисел)
    • std::cout и std::cin — тоже глобальные объекты

    Как защититься

    Если глобальная переменная всё же нужна:

    C++ — инкапсуляция через функцию-аксессор 
    // constants.cpp
namespace constants
{
    constexpr double gravity { 9.8 }; // внутренняя компоновка
}

double getGravity() // внешняя компоновка — доступна из других файлов
{
    return constants::gravity;
}

// main.cpp
#include <iostream>
double getGravity();

int main()
{
    std::cout << getGravity() << '\n'; // 9.8
    return 0;
}

    Ещё лучше — передавать значение как параметр функции, а не обращаться к глобальной напрямую. Это делает функцию переиспользуемой:

    C++ — передача через параметр (лучше) 
    namespace constants { constexpr double gravity { 9.8 }; }

// Плохо: функция жёстко привязана к глобальной
double instantVelocity(int time)
{
    return constants::gravity * time;
}

// Хорошо: функция работает с любым значением гравитации
double instantVelocity(int time, double gravity)
{
    return gravity * time;
}

int main()
{
    std::cout << instantVelocity(5, constants::gravity) << '\n';
    return 0;
}
    Лучшая практика

    Используйте локальные переменные вместо глобальных везде, где это возможно.

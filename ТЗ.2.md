# 🗺️ ROADMAP: Что делать после API Gateway

Отличный вопрос. После реализации Gateway у вас есть **стратегический выбор**: что выносить первым, чтобы максимизировать ценность для проекта и для магистерской диссертации.

Ниже — приоритизированный план с обоснованием, артефактами для ВКР и чек-листами.

## 🥇 РЕКОМЕНДУЕМЫЙ СЛЕДУЮЩИЙ ШАГ: **Auth Service**

### Почему именно аутентификация?

```
✅ Кросс-сервисная зависимость: все будущие сервисы будут требовать аутентификации
✅ Чёткие границы: легко выделить из монолита (токены, роли, сессии)
✅ Демонстрация инженерной сложности: JWT, refresh-токены, OAuth2, безопасность
✅ Измеримый результат: метрики времени входа, успешности аутентификации, атаки
✅ Безопасность как тема для ВКР: можно написать главу про "Security-by-Design в микросервисах"
```

### 📋 План реализации Auth Service (2-3 недели)

#### Неделя 1: Базовая функциональность
```bash
# 1. Создать изолированный репозиторий и решение
git clone <your-org>/repo-auth-service
dotnet new sln -n CppQuest.AuthService
dotnet new webapi -n CppQuest.AuthService --no-https -o src/CppQuest.AuthService

# 2. Добавить слои по Clean Architecture
dotnet new classlib -n Domain -o src/CppQuest.AuthService/Domain
dotnet new classlib -n Application -o src/CppQuest.AuthService/Application
dotnet new classlib -n Infrastructure -o src/CppQuest.AuthService/Infrastructure

# 3. Установить ключевые пакеты
dotnet add src/CppQuest.AuthService/CppQuest.AuthService.csproj package \
  Microsoft.AspNetCore.Authentication.JwtBearer \
  BCrypt.Net-Next \
  FluentValidation \
  MediatR \
  Npgsql.EntityFrameworkCore.PostgreSQL
```

**Минимальный контракт API:**
```csharp
// POST /api/auth/login
{ "email": "user@itmo.ru", "password": "***" }
→ { "accessToken": "jwt...", "refreshToken": "...", "expiresIn": 3600 }

// POST /api/auth/refresh
{ "refreshToken": "..." }
→ { "accessToken": "new-jwt...", "expiresIn": 3600 }

// POST /api/auth/logout
{ "refreshToken": "..." }  // Добавление в blacklist (Redis)
→ 204 No Content

// GET /api/auth/me  (требуется Bearer-токен)
→ { "id": "uuid", "email": "...", "roles": ["student"], "studentId": "M4205" }
```

#### Неделя 2: Инженерная сложность + интеграция с Gateway
| Задача | Реализация | Артефакт для ВКР |
|--------|------------|-----------------|
| **JWT с ролями** | `ClaimTypes.Role`, policy-based authorization | Схема claims, пример политики |
| **Refresh-токены в Redis** | Хранение с TTL, валидация при каждом использовании | Диаграмма потока обновления токена |
| **Rate limiting на login** | SlidingWindowLimiter для защиты от brute-force | График атак до/после защиты |
| **Интеграция с Gateway** | YARP пробрасывает `Authorization` header, валидация токена в Gateway (опционально) | Схема проверки токена на входе |
| **Метрики безопасности** | `auth_login_attempts_total{status="success|failed"}`, `auth_token_refresh_total` | Дашборд "Auth Security" в Grafana |

#### Неделя 3: Тестирование + документация
```csharp
// Пример unit-теста на валидацию пароля
[Theory]
[InlineData("weak", false)]
[InlineData("Str0ngP@ss!", true)]
public void PasswordValidator_EnforcesComplexity(string pwd, bool expected)
{
    var validator = new PasswordValidator();
    var result = validator.Validate(pwd);
    result.IsValid.Should().Be(expected);
}

// Интеграционный тест с Testcontainers
[Fact]
public async Task Login_ValidCredentials_ReturnsJwt()
{
    await using var db = await PostgreSqlContainer.Create();
    var factory = new WebApplicationFactory<Program>()
        .WithTestDatabase(db);
    
    var response = await factory.CreateClient()
        .PostAsJsonAsync("/api/auth/login", new { email = "test@itmo.ru", password = "Valid123!" });
    
    response.StatusCode.Should().Be(HttpStatusCode.OK);
    var tokens = await response.Content.ReadFromJsonAsync<TokensDto>();
    tokens.AccessToken.Should().NotBeNull();
}
```

**Артефакты для диссертации:**
- 📊 Сравнение времени аутентификации: монолит (120мс) → микросервис с Redis-кэшем (45мс)
- 🔐 Таблица реализованных мер защиты по OWASP API Security Top 10
- 📈 График: снижение успешных brute-force атак после внедрения rate limiting

## 🥈 ПАРАЛЛЕЛЬНЫЙ ТРЕК: **Shared Infrastructure & Patterns**

Пока вы делаете Auth, заложите фундамент для всех будущих сервисов — это покажет **системное мышление** в ВКР.

### Что стандартизировать:

#### 1. Шаблон проекта (Service Template)
```bash
# Создать dotnet new template в отдельном репо
dotnet new sln -n CppQuest.Microservice.Template
# ... добавить слои, middleware, Dockerfile, CI-шаблон
# Упаковать и опубликовать:
dotnet new install ./CppQuest.Microservice.Template
```

**Использование:**
```bash
# Для создания нового сервиса — одна команда:
dotnet new cppquest-microservice -n QuestService -o src/quest-service
```

#### 2. Общие middleware (в каждом сервисе)
```csharp
// CorrelationIdMiddleware — проброс сквозного идентификатора
app.UseMiddleware<CorrelationIdMiddleware>();  // Генерирует/пробрасывает X-Correlation-ID

// StructuredLoggingMiddleware — JSON-логи с контекстом
app.UseMiddleware<StructuredLoggingMiddleware>();  // Добавляет: service, version, environment

// ErrorHandlerMiddleware — единый формат ошибок
app.UseMiddleware<ErrorHandlerMiddleware>();  // Возвращает {"error": {"code", "message", "traceId"}}
```

#### 3. Health Checks & Metrics Convention
```csharp
// Единый endpoint для всех сервисов
app.MapHealthChecks("/health");        // Liveness
app.MapHealthChecks("/ready");         // Readiness (зависит от БД, Kafka и т.д.)
app.MapMetrics("/metrics");            // Prometheus endpoint (стандартный путь)
```

#### 4. Dockerfile Standardization
```dockerfile
# Multi-stage build с одинаковой структурой для всех сервисов
FROM mcr.microsoft.com/dotnet/sdk:10 AS build
WORKDIR /src
COPY *.sln *.csproj ./
RUN dotnet restore
COPY . .
RUN dotnet publish -c Release -o /app/publish

FROM mcr.microsoft.com/dotnet/aspnet:10
WORKDIR /app
COPY --from=build /app/publish .
ENTRYPOINT ["dotnet", "CppQuest.ServiceName.dll"]
```

**Артефакт для ВКР:**  
📄 Глава "Стандартизация разработки микросервисов: шаблоны, конвенции, автоматизация" с примерами кода и метриками эффективности (время создания нового сервиса: 2 дня → 2 часа).

## 🥉 ТРЕТИЙ ПРИОРИТЕТ: **Profile Service** (или User Context)

После Auth логично вынести профиль пользователя — это:
- Относительно простой CRUD + бизнес-логика (аватар, настройки, прогресс)
- Хороший полигон для демонстрации **CQRS** и **медленных/быстрых моделей чтения**
- Возможность показать **событийную интеграцию**: при регистрации → событие `UserRegistered` → профиль создаётся асинхронно

### Минимальный контракт:
```csharp
// GET /api/profile/{userId}
→ { "id": "uuid", "studentId": "M4205", "displayName": "Dmitry", "avatarUrl": "...", "stats": {...} }

// PATCH /api/profile/{userId} (требуется аутентификация + проверка прав)
{ "displayName": "New Name" }
→ 204 No Content

// GET /api/profile/me  (токен → userId из claims)
→ профиль текущего пользователя
```

### Инженерная сложность для ВКР:
| Паттерн | Реализация | Ценность для диссертации |
|---------|------------|-------------------------|
| **CQRS** | Разделение `ProfileReadDbContext` (оптимизирован для чтения) и `ProfileWriteDbContext` (для команд) | Сравнение производительности: единая БД → разделение моделей |
| **Outbox Pattern** | При изменении профиля → запись события в `outbox` таблицу → фоновая публикация в Kafka | Гарантия доставки событий, сравнение с naive pub/sub |
| **Кэширование** | Redis для частых запросов профиля, инвалидация по событиям | График: RPS до/после кэша, latency p95 |

## 🔁 АЛЬТЕРНАТИВНЫЙ ПУТЬ: Если нужна быстрая "победа"

Если комиссия скоро, и нужно показать прогресс — начните с **наблюдаемости (Observability)**, а не с нового сервиса:

### Неделя 1: Distributed Tracing
```bash
dotnet add package OpenTelemetry.Exporter.Jaeger  # или Zipkin
```
```csharp
// В Program.cs каждого сервиса
builder.Services.AddOpenTelemetry()
    .WithTracing(tracing => tracing
        .AddAspNetCoreInstrumentation()
        .AddHttpClientInstrumentation()
        .AddJaegerExporter());  // или OTLP для Prometheus+Tempo
```

**Результат:**  
🔍 В Jaeger/Tempo видно полный путь запроса:  
`Frontend → Gateway → Auth → Profile → Legacy` с таймингами на каждом хоппе.

### Неделя 2: Centralized Logging
```bash
# Вариант А: Seq (проще для демо)
dotnet add package Serilog.Sinks.Seq

# Вариант Б: Loki + Promtail (ближе к production)
```
```csharp
// Единая настройка логирования во всех сервисах
Log.Logger = new LoggerConfiguration()
    .Enrich.WithProperty("Service", "auth-service")
    .Enrich.WithProperty("Version", "1.2.0")
    .WriteTo.Console(new JsonFormatter())
    .WriteTo.Seq("http://seq:5341")
    .CreateLogger();
```

**Артефакт для ВКР:**  
📊 Кейс "Поиск причины инцидента за 2 минуты вместо 2 часов: как correlation-id + централизованные логи ускорили отладку".

## 🚀 БОНУС: Быстрый старт для следующего сервиса

Готовый скрипт `scripts/new-service.ps1`:
```powershell
param([string]$ServiceName)

# 1. Создать репозиторий (если используется GitHub CLI)
gh repo create "cppquest-$ServiceName-service" --private --confirm

# 2. Клонировать и создать структуру
git clone "git@github.com:your-org/cppquest-$ServiceName-service.git"
cd "cppquest-$ServiceName-service"

# 3. Применить шаблон (если создан)
dotnet new cppquest-microservice -n $ServiceName -o src/CppQuest.$ServiceName

# 4. Добавить базовые пакеты
dotnet add src/CppQuest.$ServiceName/CppQuest.$ServiceName.csproj package `
  MediatR, FluentValidation, Npgsql.EntityFrameworkCore.PostgreSQL, Serilog.AspNetCore

# 5. Инициализировать CI
Copy-Item "../infra/templates/ci.yml" -Destination ".github/workflows/ci.yml"

# 6. Закоммитить
git add . && git commit -m "feat: initial $ServiceName service structure" && git push
```

Использование:
```powershell
.\new-service.ps1 -ServiceName Quest
# → через 2 минуты у вас готовый репозиторий с правильной структурой
```

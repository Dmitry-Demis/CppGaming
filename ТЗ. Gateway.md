Техническое Задание (ТЗ)
Проект: Высоконагруженный API Gateway на базе YARP для миграции монолитной системы CppQuest
1. Цель проекта
Разработка интеллектуального слоя маршрутизации (API Gateway), обеспечивающего бесшовную миграцию функционала из устаревшего монолита (Legacy) в новую микросервисную архитектуру с соблюдением требований по безопасности, отказоустойчивости и глубокому мониторингу.
2. Ключевые архитектурные показатели (Магистерский уровень)
•	Strangler Fig Pattern: Постепенное «обволакивание» монолита новыми сервисами. Gateway должен уметь переключать трафик между старым и новым кодом на лету.
•	Физическая изоляция (Strict Isolation): 
o	Каждый микросервис — это отдельное решение (.sln) в собственном Git-репозитории.
o	Независимые циклы сборки и деплоя (CI/CD).
o	Собственные схемы баз данных, исключающие Shared Database антипаттерн.
•	Infrastructure as Code (IaC): Описание всей инфраструктуры (Kafka, Prometheus, Redis, базы данных) в Docker Compose для гарантированной повторяемости окружения.
3. Технические требования
3.1. API Gateway (YARP)
•	Протоколы: Поддержка HTTP/1.1, HTTP/2 и задел под gRPC.
•	Динамическая маршрутизация: Конфигурация через appsettings.json с возможностью обновления без перезагрузки (Hot Reload).
•	Трансформация запросов: Удаление/добавление заголовков авторизации и проброс X-Forwarded-For.
•	Resilience (Отказоустойчивость): Реализация паттернов Retry (повтор) и Circuit Breaker (предохранитель) через интеграцию с Polly для защиты от каскадных сбоев.
3.2. Наблюдаемость (Observability) & Метрики
Необходимо реализовать сбор и визуализацию следующих метрик:
•	IT-метрики (RED Pattern):
o	Rate (Количество запросов в секунду).
o	Errors (Количество и процент ошибок 4xx/5xx).
o	Duration (Перцентили времени отклика — p50, p95, p99).
•	Инструментарий:
o	Prometheus: Сбор метрик через /metrics эндпоинт Gateway.
o	Grafana: Построение дашбордов для мониторинга состояния системы в реальном времени.
o	Kafka: Использование как шины данных для аудита событий доступа и асинхронной передачи логов.
3.3. Безопасность (Security)
•	Rate Limiting: Ограничение количества запросов по IP или UserID для защиты от Brute-force и DoS-атак.
•	JWT Validation: Централизованная проверка подписи токена на уровне Gateway до того, как запрос попадет во внутреннюю сеть.
•	CORS Policy: Настройка политик доступа только для доверенных доменов фронтенда.
•	Security Headers: Принудительная вставка заголовков (HSTS, X-Content-Type-Options, CSP).
3.4. Тестирование
•	Unit-тесты: Покрытие логики трансформации запросов и кастомных Middleware (цель — >80% покрытия).
•	Integration Tests: Тестирование связки Gateway + Mock-сервис для проверки прохождения заголовков и обработки таймаутов.
4. Структура системы (Микросервисы)
Репозиторий / Решение	Роль	Стек	БД
CppQuest.Gateway	Входная точка, Auth check	YARP, .NET	-
CppQuest.Auth	Управление сессиями, JWT	.NET, EF Core	PostgreSQL (Users)
CppQuest.Profile	Данные профиля, прогресс	.NET, EF Core	PostgreSQL (Profiles)
CppQuest.Legacy	Монолит (старый код)	Legacy Code	Shared DB (Legacy)
5. Этапы реализации
1.	Инфраструктурный: Развертывание Docker-окружения (Kafka, Prometheus, Grafana, Postgres-инстансы).
2.	Проксирование: Создание проекта Gateway и настройка YARP для полной пересылки всех запросов на монолит (Fallback).
3.	Безопасность: Интеграция JWT-валидации и Middleware для Rate Limiting.
4.	Мониторинг: Настройка экспорта метрик в Prometheus и создание первого дашборда в Grafana.
5.	Экстракция: Создание решения Auth Service, перенос логики авторизации из монолита и перенастройка маршрута в Gateway.
6.	Верификация: Проведение нагрузочного тестирования и замер метрик доступности.
6. Критерии приемки (V-Model)
•	Система успешно обрабатывает 1000 RPS (Requests Per Second) с задержкой не более 100мс на уровне Gateway.
•	При падении одного из микросервисов Gateway корректно возвращает 503 или Fallback-ответ, не блокируя работу остальных частей.
•	В Grafana отображается график корреляции между количеством запросов и нагрузкой на CPU/RAM.

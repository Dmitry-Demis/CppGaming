# Документ требований

## Введение

API Gateway — интеллектуальный слой маршрутизации для проекта CppQuest, обеспечивающий бесшовную миграцию монолитной системы в микросервисную архитектуру по паттерну Strangler Fig. Gateway является единой точкой входа для всех клиентских запросов, централизует проверку безопасности, управляет отказоустойчивостью и обеспечивает глубокую наблюдаемость системы.

## Глоссарий

- **Gateway** — сервис CppQuest.Gateway на базе YARP (.NET), единая точка входа для всех входящих HTTP-запросов.
- **YARP** — библиотека обратного прокси (Yet Another Reverse Proxy) для .NET, используемая как ядро маршрутизации.
- **Legacy** — монолитный сервис CppQuest.Legacy, работающий на устаревшей кодовой базе.
- **Auth_Service** — микросервис CppQuest.Auth, управляющий сессиями и JWT-токенами.
- **Profile_Service** — микросервис CppQuest.Profile, хранящий данные профиля и прогресс пользователя.
- **JWT** — JSON Web Token, стандарт токена авторизации.
- **Circuit_Breaker** — паттерн отказоустойчивости, прерывающий поток запросов к недоступному сервису.
- **Retry_Policy** — политика повторных попыток при временных сбоях.
- **Rate_Limiter** — компонент ограничения частоты запросов.
- **Prometheus** — система сбора и хранения метрик.
- **Grafana** — система визуализации метрик и построения дашбордов.
- **Kafka** — распределённая шина событий для аудита и асинхронной передачи логов.
- **Redis** — in-memory хранилище, используемое для распределённого Rate Limiting.
- **Middleware** — компонент конвейера обработки HTTP-запросов в ASP.NET Core.
- **RPS** — Requests Per Second, количество запросов в секунду.
- **RED_Metrics** — метрики по паттерну RED: Rate (частота), Errors (ошибки), Duration (длительность).
- **Strangler_Fig** — архитектурный паттерн постепенной замены монолита микросервисами.
- **IaC** — Infrastructure as Code, описание инфраструктуры в виде кода.
- **Hot_Reload** — обновление конфигурации маршрутизации без перезапуска процесса Gateway.

## Требования

### Требование 1: Базовое проксирование и маршрутизация

**User Story:** Как разработчик платформы, я хочу, чтобы Gateway перенаправлял все входящие запросы на соответствующие сервисы, чтобы клиенты взаимодействовали с единой точкой входа независимо от внутренней топологии.

#### Критерии приёмки

1. THE Gateway SHALL принимать входящие HTTP/1.1 и HTTP/2 запросы на едином публичном эндпоинте.
2. WHEN запрос поступает на маршрут, сопоставленный с Auth_Service, THE Gateway SHALL перенаправить запрос в Auth_Service, сохранив метод, путь и тело запроса.
3. WHEN запрос поступает на маршрут, сопоставленный с Profile_Service, THE Gateway SHALL перенаправить запрос в Profile_Service, сохранив метод, путь и тело запроса.
4. WHEN запрос поступает на маршрут, не сопоставленный ни с одним микросервисом, THE Gateway SHALL перенаправить запрос в Legacy в качестве Fallback-маршрута.
5. THE Gateway SHALL поддерживать конфигурацию маршрутов через файл appsettings.json.
6. WHEN файл appsettings.json изменяется во время работы, THE Gateway SHALL применить новую конфигурацию маршрутизации без перезапуска процесса (Hot_Reload) в течение 5 секунд.

---

### Требование 2: Трансформация запросов и заголовков

**User Story:** Как инженер по безопасности, я хочу, чтобы Gateway управлял заголовками запросов, чтобы внутренние сервисы получали корректный контекст клиента и не раскрывали внутренние детали.

#### Критерии приёмки

1. WHEN Gateway перенаправляет запрос во внутренний сервис, THE Gateway SHALL добавить заголовок `X-Forwarded-For` с IP-адресом исходного клиента.
2. WHEN Gateway перенаправляет запрос во внутренний сервис, THE Gateway SHALL удалить исходный заголовок `Authorization` из проксируемого запроса и заменить его на внутренний заголовок с верифицированными данными пользователя.
3. THE Gateway SHALL добавлять заголовки безопасности `Strict-Transport-Security`, `X-Content-Type-Options` и `Content-Security-Policy` ко всем исходящим ответам клиенту.
4. WHEN ответ от внутреннего сервиса содержит заголовки, раскрывающие внутреннюю топологию (например, `Server`, `X-Powered-By`), THE Gateway SHALL удалить эти заголовки из ответа клиенту.

---

### Требование 3: Централизованная JWT-валидация

**User Story:** Как инженер по безопасности, я хочу, чтобы Gateway проверял JWT-токены до передачи запроса во внутреннюю сеть, чтобы ни один неавторизованный запрос не достигал микросервисов.

#### Критерии приёмки

1. WHEN запрос поступает на защищённый маршрут без заголовка `Authorization`, THE Gateway SHALL вернуть ответ со статусом 401 и не передавать запрос во внутренний сервис.
2. WHEN запрос содержит JWT с недействительной подписью, THE Gateway SHALL вернуть ответ со статусом 401 и не передавать запрос во внутренний сервис.
3. WHEN запрос содержит JWT с истёкшим сроком действия, THE Gateway SHALL вернуть ответ со статусом 401 и не передавать запрос во внутренний сервис.
4. WHEN запрос содержит валидный JWT, THE Gateway SHALL извлечь claims пользователя и передать их во внутренний сервис через внутренние заголовки.
5. THE Gateway SHALL проверять подпись JWT с использованием публичного ключа, загружаемого из конфигурации, без обращения к Auth_Service при каждом запросе.

---

### Требование 4: Rate Limiting

**User Story:** Как инженер по безопасности, я хочу ограничивать частоту запросов от одного источника, чтобы защитить систему от Brute-force и DoS-атак.

#### Критерии приёмки

1. WHEN количество запросов с одного IP-адреса превышает установленный лимит в скользящем временном окне, THE Rate_Limiter SHALL вернуть ответ со статусом 429 для последующих запросов до истечения окна.
2. WHEN количество запросов от одного UserID (из JWT) превышает установленный лимит в скользящем временном окне, THE Rate_Limiter SHALL вернуть ответ со статусом 429 для последующих запросов до истечения окна.
3. THE Rate_Limiter SHALL хранить счётчики запросов в Redis для обеспечения корректной работы в распределённой среде с несколькими экземплярами Gateway.
4. THE Gateway SHALL включать заголовок `Retry-After` в ответы со статусом 429, указывающий время в секундах до сброса лимита.
5. WHERE маршрут помечен как публичный в конфигурации, THE Rate_Limiter SHALL применять отдельный, более высокий лимит запросов для анонимных клиентов.

---

### Требование 5: Отказоустойчивость (Resilience)

**User Story:** Как SRE-инженер, я хочу, чтобы Gateway корректно обрабатывал сбои внутренних сервисов, чтобы частичная недоступность одного сервиса не приводила к полному отказу системы.

#### Критерии приёмки

1. WHEN внутренний сервис не отвечает в течение 3 секунд, THE Gateway SHALL прервать соединение и вернуть клиенту ответ со статусом 504.
2. WHEN запрос к внутреннему сервису завершается ошибкой сети, THE Retry_Policy SHALL повторить запрос не более 3 раз с экспоненциальной задержкой перед возвратом ошибки клиенту.
3. WHEN количество последовательных ошибок от одного внутреннего сервиса превышает 5 в течение 30 секунд, THE Circuit_Breaker SHALL перейти в состояние «открыт» и немедленно возвращать ответ со статусом 503 без обращения к сервису.
4. WHILE Circuit_Breaker находится в состоянии «открыт», THE Gateway SHALL периодически (каждые 10 секунд) отправлять один пробный запрос для проверки восстановления сервиса.
5. WHEN пробный запрос Circuit_Breaker завершается успешно, THE Circuit_Breaker SHALL перейти в состояние «закрыт» и возобновить нормальную маршрутизацию.
6. IF внутренний сервис недоступен и для маршрута определён Fallback, THEN THE Gateway SHALL перенаправить запрос на Legacy вместо возврата ошибки.

---

### Требование 6: Наблюдаемость — сбор метрик (RED Pattern)

**User Story:** Как SRE-инженер, я хочу собирать метрики по паттерну RED для каждого маршрута, чтобы иметь полную картину состояния системы в реальном времени.

#### Критерии приёмки

1. THE Gateway SHALL экспортировать метрику `gateway_requests_total` (счётчик) с метками `route`, `method`, `status_code` через эндпоинт `/metrics` в формате Prometheus.
2. THE Gateway SHALL экспортировать метрику `gateway_request_duration_seconds` (гистограмма) с перцентилями p50, p95, p99 с метками `route` и `method`.
3. THE Gateway SHALL экспортировать метрику `gateway_errors_total` (счётчик) с метками `route` и `error_type` для всех ответов с кодами 4xx и 5xx.
4. WHEN Prometheus опрашивает эндпоинт `/metrics`, THE Gateway SHALL вернуть актуальные метрики в формате OpenMetrics в течение 200 мс.
5. THE Gateway SHALL экспортировать метрику `circuit_breaker_state` (gauge) с метками `service`, отражающую текущее состояние Circuit_Breaker для каждого внутреннего сервиса.

---

### Требование 7: Наблюдаемость — аудит событий через Kafka

**User Story:** Как специалист по безопасности, я хочу, чтобы все события доступа записывались в аудит-лог через Kafka, чтобы иметь возможность расследовать инциденты и анализировать паттерны доступа.

#### Критерии приёмки

1. WHEN запрос успешно проксируется, THE Gateway SHALL опубликовать событие аудита в топик Kafka `gateway.access.events` с полями: timestamp, client_ip, user_id, route, method, status_code, duration_ms.
2. WHEN JWT-валидация завершается неудачей, THE Gateway SHALL опубликовать событие безопасности в топик Kafka `gateway.security.events` с полями: timestamp, client_ip, failure_reason.
3. THE Gateway SHALL публиковать события в Kafka асинхронно, не блокируя обработку запроса.
4. IF Kafka недоступна, THEN THE Gateway SHALL продолжить обработку запросов и записать пропущенные события в локальный буфер для последующей отправки.

---

### Требование 8: CORS-политика

**User Story:** Как разработчик фронтенда, я хочу, чтобы Gateway корректно обрабатывал CORS-запросы, чтобы браузерные клиенты с доверенных доменов могли взаимодействовать с API.

#### Критерии приёмки

1. WHEN браузер отправляет preflight OPTIONS-запрос с заголовком `Origin` из списка доверенных доменов, THE Gateway SHALL вернуть ответ со статусом 204 и корректными CORS-заголовками.
2. WHEN запрос поступает с `Origin`, не входящим в список доверенных доменов, THE Gateway SHALL вернуть ответ без CORS-заголовков, что приведёт к блокировке браузером.
3. THE Gateway SHALL загружать список доверенных доменов из конфигурации appsettings.json с поддержкой Hot_Reload.

---

### Требование 9: Инфраструктура как код (IaC)

**User Story:** Как DevOps-инженер, я хочу описать всю инфраструктуру в Docker Compose, чтобы любой разработчик мог поднять полное окружение одной командой.

#### Критерии приёмки

1. THE IaC SHALL описывать все инфраструктурные компоненты (Kafka, Prometheus, Grafana, Redis, PostgreSQL-инстансы для Auth_Service и Profile_Service) в едином файле `docker-compose.yml`.
2. WHEN выполняется команда `docker compose up`, THE IaC SHALL запустить все сервисы в корректном порядке с учётом зависимостей (health checks).
3. THE IaC SHALL монтировать конфигурационные файлы Prometheus и Grafana из репозитория, обеспечивая воспроизводимость дашбордов.
4. THE IaC SHALL использовать именованные Docker volumes для персистентного хранения данных PostgreSQL и Kafka, исключая потерю данных при перезапуске контейнеров.

---

### Требование 10: Производительность

**User Story:** Как архитектор системы, я хочу, чтобы Gateway обрабатывал высокую нагрузку с минимальными задержками, чтобы Gateway не становился узким местом системы.

#### Критерии приёмки

1. WHILE Gateway обрабатывает нагрузку в 1000 RPS, THE Gateway SHALL поддерживать медианное время отклика (p50) не более 100 мс на уровне Gateway (без учёта времени ответа внутренних сервисов).
2. WHILE Gateway обрабатывает нагрузку в 1000 RPS, THE Gateway SHALL поддерживать p99 время отклика не более 500 мс.
3. THE Gateway SHALL использовать асинхронную обработку запросов (async/await) во всех Middleware для предотвращения блокировки потоков.

---

### Требование 11: Тестирование

**User Story:** Как разработчик, я хочу иметь автоматизированные тесты для Gateway, чтобы изменения в логике маршрутизации и безопасности не приводили к регрессиям.

#### Критерии приёмки

1. THE Test_Suite SHALL содержать unit-тесты для каждого кастомного Middleware (JWT-валидация, Rate Limiting, трансформация заголовков) с покрытием не менее 80% строк кода.
2. THE Test_Suite SHALL содержать интеграционные тесты, проверяющие сквозное прохождение запроса через Gateway к Mock-сервису с верификацией корректности заголовков в проксируемом запросе.
3. THE Test_Suite SHALL содержать интеграционные тесты, проверяющие поведение Gateway при таймауте внутреннего сервиса (возврат 504) и при срабатывании Circuit_Breaker (возврат 503).
4. FOR ALL конфигураций маршрутизации, сериализованных в appsettings.json, THE Test_Suite SHALL верифицировать, что десериализация и повторная сериализация конфигурации производит эквивалентный объект (round-trip свойство).

---

### Требование 12: Документация кода и README

**User Story:** Как разработчик, входящий в проект, я хочу, чтобы весь публичный код был задокументирован на русском языке в стандарте Doxygen, а проект содержал README с инструкциями по запуску и проверке, чтобы я мог быстро разобраться в архитектуре и начать работу.

#### Критерии приёмки

1. ALL публичные классы, интерфейсы, record-типы и перечисления SHALL содержать XML-комментарий `/// <summary>` на русском языке с описанием назначения типа.
2. ALL публичные свойства SHALL содержать XML-комментарий `/// <summary>` на русском языке с описанием смысла хранимого значения.
3. ALL публичные методы SHALL содержать XML-комментарии `/// <summary>`, `/// <param name="...">` для каждого параметра и `/// <returns>` при наличии возвращаемого значения; `/// <exception cref="...">` — при наличии задокументированных исключений.
4. THE Project SHALL содержать файл `README.md` с разделами: структура проекта, требования к окружению, быстрый старт через Docker Compose, запуск только Gateway локально, запуск тестов, проверка работоспособности (примеры curl), описание конфигурации.
